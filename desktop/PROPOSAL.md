# KerFlow — Product Proposal

## Summary

KerFlow 是一个独立的桌面端产品,服务于个人的需求管理全流程。从一个想法的诞生,到讨论、拉分支、写代码、写提案、提 PR、Review,全部在一个应用内完成。

核心理念:把 kerminal-proposal-workflow 里的每一步都变成可视化的、可操作的界面,用户不需要在终端、编辑器、浏览器之间来回切换。

## Motivation

当前的工作流依赖多个工具拼接:

- 想法记录:靠记忆或散落的笔记
- 需求状态:去 GitHub 网页看 PR
- 执行流程:在终端里跟 Kerminal 对话,手动 git 操作
- 进度跟踪:靠人脑记住走到哪一步了

KerFlow 要把这些合成一个产品,让"从想法到 PR"成为一条清晰、可视、不断档的流水线。

## Users / Scenarios

### 用户:产品负责人(Juno)

日常同时推进多个需求,每个需求处于不同阶段。希望:

- 随时记录冒出来的想法,不丢失
- 一眼看到所有需求的当前状态
- 想启动一个需求时,直接在 App 里跟 Kerminal 沟通,不用切到终端
- 全流程在一个地方完成:拉分支 → 讨论实现 → 改代码 → 写提案 → 提 PR → 看 Review 结果

## Requirements

### 1. 想法池

- 顶部输入框,回车即新增一条想法。
- 想法累积成列表,按创建时间倒序。
- 每条想法可编辑、可删除。
- 每条想法有两种状态:
  - 💡 草稿 — 还没想清楚
  - 🚀 待启动 — 想清楚了,准备走 workflow
- 草稿在上,待启动在下(待启动意味着即将变成正式需求)。
- 输入即保存,关闭重开不丢。

### 2. 需求看板

- 自动从 GitHub 同步用户创建的 PR 状态。
- 按阶段分组展示:开发中 / 提案中 / Review 中 / 已合入 / 已关闭。
- 每张卡片展示:需求名称、PR 编号、分支名、更新时间、review 数。
- 点击卡片可查看详情(PR body)或跳转 GitHub。
- 定时轮询保持最新。

### 3. Workflow 全流程(从想法到 PR)

当用户对一条"待启动"的想法点击"启动需求",进入 workflow 流程。参照 kerminal-proposal-workflow.md,每一步在 App 内可视化呈现:

**阶段 0 · 选基准分支**
- App 列出候选分支(main / 相关旧分支),用户选择。

**阶段 1 · 拉取 + 构建**
- App 执行 git checkout + pull,显示进度。
- 本地构建验证,展示结果(成功/失败)。

**阶段 2 · 讨论 + 实现**
- 内嵌 Kerminal 对话界面。
- 用户描述要实现的功能,跟 Kerminal 讨论。
- Kerminal 改代码,App 展示改动文件列表与验证结果(typecheck / lint)。

**阶段 3 · 写内部 PRD**
- Kerminal 起草 proposal 大纲,用户确认后填细节。
- App 内可预览 markdown 渲染效果。

**阶段 4 · 转成 Git 规范提案**
- 按标准模板结构化提案。
- 存放到 `product/proposals/<slug>/proposal.md`。

**阶段 5 · 提交 PR**
- Kerminal 准备 commit + PR 标题/Body。
- 用户确认后,App 执行 push 并创建 PR。
- 展示 PR 链接。

**阶段 6 · Review**
- App 拉取 GitHub Review 结果(bot 评审 + 人工 review)。
- 展示通过/未通过及具体建议。
- 用户决定是否采纳 → 回到阶段 2 迭代。

### 4. 内嵌 Kerminal 对话

- App 内集成 Kerminal 对话界面,类似 KerWork 的会话体验。
- 每个需求对应一个独立的 Kerminal 会话,上下文隔离。
- 支持实时流式输出、执行计划展示、权限审批。
- Kerminal 的本地文件操作、git 操作在 App 内可见。

## Constraints

1. 独立产品,不依赖 KerWork,不嵌入 KerWork。
2. 仅服务个人使用,不考虑多人协作。
3. 需要本地 Kerminal 二进制(同 KerWork 的 sidecar 模式)。
4. 需要 GitHub 访问权限(OAuth 或 token)。
5. 目标仓库可配置。
6. 技术架构可参考 KerWork(Tauri + SolidJS),但独立仓库、独立打包。

## Current State

- 当前无此产品,workflow 依赖终端 + GitHub 网页 + 人脑串联。
- kerminal-proposal-workflow.md 定义了完整流程,但纯靠人工执行。
- 刚完成的 pr-board 网页版验证了看板 + GitHub 同步的可行性,但受限于网页无法操作本地,决定改为桌面端。

## Product Proposal

### 产品定位

KerFlow = 个人需求工作台 + Kerminal 驱动的 workflow 引擎。

一句话:从想法到 PR,一个 App 搞定。

### 技术架构

- 框架:Tauri + SolidJS(参考 KerWork)
- AI 后端:Kerminal sidecar(同 KerWork 方式)
- 数据存储:本地文件(想法池 JSON + Kerminal rollout)
- GitHub 集成:OAuth token,调 REST API
- Git 操作:通过 Kerminal 或直接调 git CLI

## Acceptance Criteria

### 想法池
- 能输入并保存想法,关闭重开不丢失。
- 能编辑、删除、标记状态(草稿/待启动)。

### 需求看板
- 能看到自己在目标仓库的所有 PR,按阶段分组。
- 点击能看到详情,能跳转 GitHub。
- 定时自动刷新。

### Workflow 全流程
- 从"待启动"点击"启动需求"能进入 workflow。
- 每个阶段可视化展示进度。
- 能在 App 内完成拉分支、跟 Kerminal 对话、写提案、提 PR 的全流程。
- PR 创建后自动出现在需求看板中。

### Kerminal 对话
- App 内能跟 Kerminal 正常对话,流式输出。
- 每个需求独立会话,上下文不串。
- 权限审批在 App 内完成。
